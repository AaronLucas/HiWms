# ADR-014: 唯一追踪策略三层解析 + 无码/未识别货物分离处理

## 状态
✅ Accepted (2026-07-16) —— **全新设计，本地零实现，迁移脚本尚未起草**

## 背景

仓库现场普遍存在"货物无法扫码"的情况，但这背后是两件完全不同的事：
1. 低值货物本来就没打算单独追踪——如果系统看到"无码"就一律报异常，操作员会觉得系统在无理由增加工作量。
2. 本该有码但现场缺失/损坏——如果这种情况被悄悄放行，高值货物的追溯链条就断了。

混在一起处理，两种问题都会恶化。DBA 团队据此提出：**判断顺序应该是先问"这个商品/这个库位，策略上要不要唯一追踪"，只有"策略要求追踪但现场恰好没有码"才是真正的异常**；策略本来就不要求追踪的，无码是正常状态，不触发任何异常。

## 决策

### 1. 追踪策略走三层解析，具体覆盖笼统
优先级从高到低：商品级覆盖（`product_constraints.requires_unique_tracking`）> 库位级强制（`locations.force_unique_tracking`，**只能调严不能调松**——不支持"库位让本该追踪的商品免于追踪"，那等于让物理分区绕过商品风控）> 租户按 ABC 分类的默认值（`tenant_tracking_policies`）。

**B 类商品刻意不提供系统默认值**——这是本轮讨论明确达成的共识：B 类默认策略是租户的运营决策，系统不该代为拍板。未配置时保守按"需要追踪"处理，且必须在部署清单里提醒租户必须显式配置，不能当作长期方案。

### 2. 两条完全不同的异常路径，而不是合并成一种"无码异常"
- **`MISSING_LABEL`**（商品身份明确，只是缺容器身份）：先按数量正常记账（不阻塞后续正常出库分配），登记异常等待补码；补码闭环复用统一异常领域的审计轨迹机制（生成内部码 → 物理贴码 → 扫码确认匹配 → 挂载容器 → 关闭异常），不是单独发明一套状态机。
- **`UNIDENTIFIED_GOODS`**（连商品身份都无法确定，严重度 HIGH，高于 MISSING_LABEL 的 MEDIUM）：先暂存（`product_id=NULL`），登记异常，等待人工识别回填。

### 3. 合规触发器范围必须扩展
既有的 `fn_trg_enforce_product_constraints`（V2.1）原本只在 `BEFORE UPDATE OF location_id` 触发。但"识别未识别货物、回填 product_id"这个动作只改 `product_id`，不改 `location_id`——原范围下触发器根本不会重新跑，等于合规校验被静默跳过：一批货物可能已经在不合规库位待了很久，系统却毫无察觉。这是本轮测试中发现的真实 bug，不是理论风险。决策：把触发器范围扩展为 `BEFORE INSERT OR UPDATE OF location_id, product_id`。

## 后果

### 正面
- 低值货物不再被无理由地要求扫码，减少操作员对系统的抵触。
- 高值/合规敏感货物的追溯链条不会被"无码就放行"悄悄打断。
- 识别未知货物时能重新触发合规校验，避免"先斩后奏"式的静默违规。

### 负面 / 风险
- **本次仅是设计决策记录**：本地没有任何对应的迁移脚本草稿（不同于 Layer 3，Layer 3 至少有一份含 bug 的本地草稿），需要用户先与 DBA 团队完成协调确认才能起草（详见 `docs/00-project/ROADMAP.md` Phase 1.4.2 的阻塞提示）。
- **部署顺序是硬约束，不是建议**：本层必须严格排在 Layer 3（ADR-013）之后部署，因为本层会用 `CREATE OR REPLACE` 重新定义 Layer 3 的 `fn_apply_putaway_action`，颠倒顺序会导致追踪策略判断被静默覆盖，且**不会有任何 SQL 报错**——这是 DBA 文档自身版本修订（v1.2→v1.3）里记录的真实教训，此前的版本错误地认为两层可以并行/任意顺序部署。
- 部署前必须完成的租户配置（B 类追踪策略、哪些库位需要 `force_unique_tracking`）都是运营决策，本 ADR 只提供机制和安全默认值，不替业务/合规拍板。
